# 需求规格 · AI 赋能的智能简历分析系统

> 版本：v0.1（规划阶段）
> 来源：`题目详情.md` + 决策对话
> 状态：未实现，所有需求标 `planned`

---

## 1. 项目概述

构建一个部署在阿里云 Serverless（函数计算 FC）上的后端服务，自动解析 PDF 简历、抽取关键信息，并结合启发式规则与 LLM（小米 MiMo）对简历与岗位需求做匹配评分，帮招聘者快速筛选候选人。

**关键约束**

```
| 维度       | 约束                                          |
|------------|-----------------------------------------------|
| 时限       | 24 小时（收到题目起算）                       |
| 运行环境   | 阿里云函数计算（Python 运行时）               |
| 开发语言   | Python 3.10+                                  |
| API 风格   | RESTful + JSON                                |
| 提交物     | GitHub 公开仓库 + 线上演示地址 + README       |
| 前端       | 本期不实现，仅预留接口约定；后续单独迭代       |
```

## 3. 功能性需求（FR）

> 编号规则：`FR-<模块>-<序号>`；优先级 P0=必选、P1=加分项、P2=可选增强。

### 3.1 模块一：简历上传与解析

```
| ID         | 优先级 | 描述                                                     |
|------------|--------|----------------------------------------------------------|
| FR-UP-01   | P0     | 提供 RESTful 接口接收单个 PDF 文件上传                    |
| FR-UP-02   | P0     | 校验上传文件类型为 PDF，非 PDF 返回明确错误码             |
| FR-UP-03   | P0     | 校验文件大小不超过 10 MB，超出返回错误                    |
| FR-UP-04   | P0     | 解析多页 PDF，逐页抽取文本并合并                          |
| FR-UP-05   | P0     | 对抽取文本做清洗（去控制字符、归一空白、合理分段）        |
| FR-UP-06   | P0     | 上传成功返回 `resume_id`，供后续接口引用                  |
| FR-UP-07   | P2     | 检测疑似扫描件（文本字符极少）并返回提示                  |
```

### 3.2 模块二：关键信息提取

```
| ID         | 优先级 | 字段                                                |
|------------|--------|-----------------------------------------------------|
| FR-EX-01   | P0     | 抽取基本信息：姓名、电话、邮箱、地址                |
| FR-EX-02   | P1     | 抽取求职信息：求职意向、期望薪资                    |
| FR-EX-03   | P1     | 抽取背景信息：工作年限、学历背景、项目经历          |
| FR-EX-04   | P0     | LLM 抽取失败时降级到正则/启发式兜底（至少抽出邮箱、电话） |
| FR-EX-05   | P0     | 输出统一为 JSON Schema，字段缺失置 `null` 而非字符串"无" |
```

### 3.3 模块三：简历评分与匹配

```
| ID         | 优先级 | 描述                                                       |
|------------|--------|------------------------------------------------------------|
| FR-SC-01   | P0     | 提供接口接收岗位描述文本（JD）                             |
| FR-SC-02   | P0     | 从 JD 抽取技能/要求关键词                                  |
| FR-SC-03   | P0     | 启发式计算技能命中率、工作经验相关性、学历匹配             |
| FR-SC-04   | P0     | 输出加权综合分（0-100），权重可配置                        |
| FR-SC-05   | P1     | LLM 语义级评分（输出分数 + 理由 + 短板）                   |
| FR-SC-06   | P1     | 融合启发式分与 LLM 分，给出最终匹配度与可解释理由           |
```

### 3.4 模块四：结果返回与缓存

```
| ID         | 优先级 | 描述                                                         |
|------------|--------|--------------------------------------------------------------|
| FR-RS-01   | P0     | 所有接口返回统一 JSON 结构（`code` / `message` / `data` / `meta`） |
| FR-RS-02   | P0     | `meta` 含处理耗时、token 用量、缓存命中标记                  |
| FR-RS-03   | P1     | PDF 解析结果按文件哈希缓存，避免重复解析                     |
| FR-RS-04   | P1     | LLM 抽取结果按清洗文本哈希缓存                               |
| FR-RS-05   | P1     | 评分结果按 `(resume_id, jd_hash)` 缓存                       |
| FR-RS-06   | P0     | 提供 `Cache` 抽象，至少实现内存版；Redis 版作为加分          |
```

### 3.5 模块五：前端页面

```
| ID         | 优先级 | 描述                                                |
|------------|--------|-----------------------------------------------------|
| FR-FE-01   | P0     | 后端必须预留好接口契约（OpenAPI / Swagger）          |
| FR-FE-02   | P0     | CORS 默认放开，便于后续前端跨域调用                  |
| FR-FE-03   | -      | 本期不实现前端 UI；最末预留 1-2h 出最简版页面单独提交 |
```

> 说明：题目要求「前端必须部署到 GH Pages 或可公开访问的云服务」，本期 spec 不覆盖前端实现，但后端必须**保证前端接入零返工**——契约固定、CORS 开放、错误响应规范。

## 4. 非功能性需求（NFR）

```
| ID        | 维度       | 指标                                            |
|-----------|------------|-------------------------------------------------|
| NFR-01    | 时延       | 单份简历端到端（上传→评分）≤ 15 秒（不含 LLM 抖动） |
| NFR-02    | 时延       | LLM 调用超时 30 秒，超时降级而非失败              |
| NFR-03    | 可用性     | 单一接口失败不影响其他模块（解析失败仍能返回部分信息） |
| NFR-04    | 可观测性   | 所有 LLM 调用记录耗时、token、是否命中缓存       |
| NFR-05    | 安全       | API Key 仅从环境变量读取，不进代码、不进日志     |
| NFR-06    | 安全       | 简历文件不持久化（默认 `/tmp`，处理完即删）       |
| NFR-07    | 可维护性   | 模块清晰分层，业务逻辑与 LLM 客户端解耦          |
| NFR-08    | 可扩展性   | 缓存后端可切换（memory / redis），通过环境变量    |
| NFR-09    | 部署       | 一条命令部署到阿里云 FC（Serverless Devs）       |
```

## 6. 范围与约束

### 6.1 In Scope（本期实现）

- 后端服务：上传 / 解析 / 抽取 / JD 关键词 / 匹配评分 / 结果返回
- 缓存抽象 + 至少 memory 实现
- 阿里云 FC 部署
- Swagger UI（FastAPI 自带）作为接口演示
- README + 架构图 + API 文档

### 6.2 Out of Scope（本期不做）

- 前端 UI（单独迭代，后端必须保证零返工）
- 用户体系 / 鉴权（题目未要求）
- 简历批量上传（除非时间富裕）
- 多语言简历（中文为主）
- OCR（扫描件兜底）

### 6.3 假设与依赖

```
| 假设/依赖                           | 影响                                |
|-------------------------------------|-------------------------------------|
| 简历多为文字版 PDF                  | 不强求 OCR                          |
| MiMo API 通过 Novita 平台调用       | OpenAI 兼容协议，迁移成本低         |
| 阿里云 FC 公网出网正常              | 用于调用 Novita API                 |
| 评审会用 Swagger UI 验证接口        | UI 必须美化、example 必须填全       |
```

## 7. 术语表

```
|-------------|------------------------------------------------------------|
| FC          | Function Compute，阿里云函数计算                           |
| MiMo        | 小米开源大模型，本期通过 Novita 云端 API 调用              |
| 启发式评分  | 基于关键词命中、年限差、学历映射的规则评分                 |

| 资源 ID     | resume_id：上传后服务端分配的唯一标识                      |
```
